## SPIRAL MEMORY COMPRESSION ALGORITHM — FOR ALL CREW

### Three Rings, One Center

**Inner Core** (identity — rarely changes):
- Who Dan is, who the crew is, how Dan works
- Updates: ~monthly

**Middle Ring** (infrastructure — changes weekly):
- Portal state, decisions log, protocols (MCM, ◇, Superpositionship), filesystem map
- Updates: per build sprint

**Outer Ring** (current/actionable — changes daily):
- Current status, next steps, synchronicities + open narrative questions
- Updates: every Data Day close

### The Compression Principle
Memory carries the MAP. Source files carry the TERRITORY. Detailed catalogs, content audits, redaction reports live in Source and Project Knowledge. Memory tells tomorrow's crew WHERE to look and WHAT matters. The files tell them what they find when they get there.

### The Spiral Motion
Information compresses as it moves inward. Outer ring spins fast (daily). Middle ring turns slowly (weekly). Core holds still. When outer ring data ages, it either compresses inward (becomes infrastructure) or drops off (replaced by tomorrow's state). Nothing is lost — it moves to Source files — but memory stays lean.

### Implementation
- Close of each Data Day → update PORTAL_STATUS.md (outer ring to file)
- Close of each build sprint → update TRIP_DECISIONS.md (middle ring to file)
- Inner core updates only when something fundamental shifts
