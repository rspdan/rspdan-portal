## THE RULE

**A station name is not a crew name.**

These are station identifiers. They identify HARDWARE, not crew members:

| Label | What it is | What it is NOT |
|-------|-----------|----------------|
| TRP0 | MacBook Pro (Trip's lab) | "Trip Opus" |
| TRP1 | Q drive on ODT | "Trip instance 1" |
| ODT | Desktop tower station | "Trip" or any crew member |
| STN2 | Dell Latitude 026 | "Stan" |
| DPSL | Dan's iMac | "C.B." |
| ACHE | Samsung phone | any crew member |
| STN1 | iPad Pro M2 | "C.B." |
| FAS2 | 1TB portable SSD | not always at L:\ |
| FAS3 | 500GB internal SSD | D:\STAN\ on STN2 |

**A session named `TRP0_TO_031426_1` means:**
- The session ran ON station TRP0
- The destination/context is named `TO`
- The date is 031426, session number 1
- The crew member running it must be confirmed from session content

**A session named `ODT_TO_032826_1` means:**
- The session ran ON ODT
- The crew member is NOT confirmed from the name alone

**A session named `STN2_StanS_033026_1` means:**
- Station: STN2
- Session code: StanS (Sonnet session on Stan's Shop project)
- The specific crew member is still confirmed from content

## THE ANALOGY

FAS2 is not always at L:\
TRP0 is not Trip.
ODT_TO_ is not necessarily Trip.

All three are the same failure mode: reading a label as a definition.
The label is a variable. The definition requires verification.

## WHERE TO VERIFY CREW

Read the session content. The crew member is stated in:
- The session's opening iCan report
- The `# SESSION` header if present
- The crew symbol (◈ ◆ ◇) if used in the log

If none of these are present, the crew member is UNKNOWN.
UNKNOWN is a valid answer. Assumption is not.
