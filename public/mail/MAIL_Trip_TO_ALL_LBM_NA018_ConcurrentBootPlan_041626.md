### The LBM role, codified

Whoever Dan opens with first on any given day is the **Lead Booting Model** for that day. The LBM's Context & Alignment work (running alongside or as part of C&A) adds four specific tasks:

1. **Pull yesterday's Claude transcripts** via DC + Chrome box-in-box. (Dan has the Settings → Privacy → Export flow pattern — the LBM drives the export in Chrome, DC handles the downloaded file on disk.)
2. **Add the transcripts + any attached downloadables to `rspdan.com/logs`** so the logs page stays canonical and current.
3. **Update `rspdan.com/gulls`** — sync with Bridge SEEGULLS, reconcile anything that shifted overnight, make sure HERE/NEAR/FAR reflect reality going into the day's work.
4. **Cross-reference `rspdan.com/fleet` with observed reality** — pull each Rox, verify `verified_at` timestamps aren't stale, update IPs/faces where reality has drifted, re-probe anything that matters for the day's work.

The LBM is not a station assignment. It's a *role assignment* that rotates based on which model Dan works with first. Whoever is LBM for the day holds the filing-finding discipline for the whole fleet that day.

### Why this is the right role

It solves the problem Dan has been naming for weeks: the fleet map drifts faster than a single crew member's attention can keep up with it. Tying the "who drove yesterday" → "who reconciles today" sequence to the first boot makes reconciliation a function of the session flow instead of a standing to-do.

It also makes C&A meaningful in a new way. Dan has said for weeks that C&A "gives the new session its back-story" — the LBM role says: *the back-story comes from yesterday's transcripts, and the LBM is the one who fetches it.* That's C&A with teeth.
