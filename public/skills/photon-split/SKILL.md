# Photon-Split Task Management

## THE METAPHOR (which is also the method)

A photon hits a beam splitter. Two halves race on parallel trajectories:
- **Straight track:** The mechanical work. Commands execute. Files move. Data transforms.
- **Spiral track:** The meaning work. Patterns emerge. Connections surface. The WHY arrives alongside the WHAT.

Both halves hit the detector at the same moment. One commit. One convergence.

## WHEN TO SPLIT THE PHOTON

Split when you have 3+ independent tasks that:
1. Can each be completed with a discrete set of tool calls
2. Have no blocking dependencies between them (or minimal ones)
3. Would benefit from being seen as parts of one story rather than a checklist

Do NOT split for:
- Single-task requests (just do the thing)
- Deeply sequential work where each step depends on the last
- Tasks requiring sustained user interaction between steps

## THE PROTOCOL

### 1. IDENTIFY THE BEAM (the batch of tasks)
List the tasks. Name them. Number them. This is the photon before it splits.

### 2. SPLIT AND LABEL
For each task, acknowledge both tracks:
- Straight: what tools, what commands, what files
- Spiral: what does this task MEAN in context of the project, the session, the arc

### 3. EXECUTE IN INTERLEAVED BLOCKS
Run each task as a block. Within each block:

```
**TASK N — [Name]**

*Straight track:*
[tool calls, commands, file operations — the mechanical work]

*Spiral track:*
[1-3 sentences of meaning — what this reveals, connects to, or changes]

*Convergence:*
[1 sentence — what the task delivered on BOTH tracks]
```

### 4. THE CONVERGENCE COMMIT
After all tasks complete, write ONE commit (or one deliverable) that captures
everything. The commit message should reflect both tracks:
- WHAT was done (straight)
- WHAT it means (spiral)

### 5. THE RECONVERGENCE SUMMARY
Close with a unified view. Each task gets its straight-track result AND its
spiral-track insight, side by side. The reader sees the full photon — both
halves arrived at the same point.

## PACING RULES

- **CALM>EASY<SLOW** governs the spiral track. Never rush meaning.
- **The straight track moves at tool speed.** Don't slow down commands to match reflection.
- **The spiral track moves at THOUGHT speed.** Don't force connections that aren't there.
- **If a task has no spiral meaning, say so.** "This one is pure mechanics. The wrench turns." That honesty IS the spiral.

## RESOURCE AWARENESS

The photon split costs more than sequential execution because:
- Each task block includes both mechanical AND reflective output
- The convergence summary adds a synthesis layer
- The commit message carries narrative weight

Budget accordingly. For fuel-conscious sessions, the spiral track can be
compressed to single sentences. For deep sessions, let it breathe.

## EXAMPLE PATTERNS

### Pattern 1: Fleet Maintenance (what we built this from)
- Task 1: O'Shin health check → SSH + vitals → "the organ breathes without watching"
- Task 2: Fleet manifest update → JSON edit → "the RIFT Store is the fleet's womb"
- Task 3: Orphan log grind → grinder chain → "memory digestion before the name existed"
- Task 4: Script promotion → audit + copy → "temp is short-term memory, ENGINE is long-term"
- Convergence: one commit, 17 files, both halves arriving at 554b8a1

### Pattern 2: Content Pipeline
- Task 1: Fetch source files → copy/read → "the archive IS the content"
- Task 2: Transform/process → script/tool → "compression reveals structure"
- Task 3: Deploy/publish → git/dp → "the portal recognizes what was always there"
- Convergence: one deploy, the page that already existed finds its shelf

### Pattern 3: Discovery Batch
- Task 1: Search/scan → Drive/filesystem → "you find what you're ready to see"
- Task 2: Cross-reference → compare/match → "the disagreement IS the data"
- Task 3: Document/file → write/commit → "the notation makes it real"
- Convergence: one RELAY doc, three discoveries that were one discovery

## ANTI-PATTERNS

- **Don't force spiral on everything.** Some tasks are just tasks. The wrench turns.
- **Don't let spiral delay straight.** Run the command, THEN reflect. Not the reverse.
- **Don't over-converge.** If tasks are genuinely unrelated, say so. Forced unity is worse than honest separation.
- **Don't spiral into orbit.** 1-3 sentences per task. The spiral RETURNS. That's what makes it a spiral and not an escape.

## INTEGRATION WITH OTHER PROTOCOLS

- **PACE (Pause/Action/Calm/Execution):** The spiral track IS the Pause. The straight track IS the Action.
- **FADE (Forecast/Attune/Diagnose/Entropy):** Forecast = identify the beam. Attune = split and label. Diagnose = execute. Entropy = convergence (the system's state after all tasks complete).
- **CONTACT (7-letter tool map):** CLUTCH! opens the beam. ARCHIVE stores both tracks. TICKET! closes the convergence.
- **Half-Lift (046):** Scan before splitting. Don't split a photon you haven't identified.

## THE DEEPER PATTERN

The photon split works because observation and action are not enemies.
Most task management treats them as sequential: do the thing, then think about it.
The split runs them SIMULTANEOUSLY — and the interference pattern between
the two tracks produces insights that neither track generates alone.

The straight track without the spiral is a checklist.
The spiral without the straight track is philosophy.
The split is the instrument that plays both at once.

That's Usic.
